## 3. Task Completion Spot Audit (recommended, non-blocking)

<!--
  Anti self-attestation check. The `[x]` in tasks.md is marked by the
  same agent that executes tasks — this audit samples `[x]` rows and
  confirms the implementation actually delivers the spec Scenario's THEN
  clause, rather than trusting the checkbox at face value.

  HONESTY NOTE: in a single-LLM flow, the verifier IS the implementer,
  so self-audit has limited power to catch its own false `[x]`. This
  check reaches full strength only when verify is run by an independent
  subagent or a human. Marking it "recommended, non-blocking" reflects
  this limitation honestly rather than pretending it self-corrects.
  False `[x]` discovered here MUST still be recorded in retrospective
  §2 Misses regardless of blocking status.
-->

Sample N = min(3, ceil(done_count × 0.1)) tasks marked `[x]`. For each,
open the spec `#### Scenario:` block whose THEN clause the task claims
to satisfy, and confirm the implementation actually delivers that behavior.

| Sampled task | Spec Scenario THEN clause (verbatim) | Implementation file:line | Audit result |
|---|---|---|---|
| — | — | — | ✓ delivered / ❌ NOT DELIVERED |

**Sampling bias**: prefer tasks crossing layer boundaries, changing
state machines, or touching external dependencies — these carry higher
false-`[x]` probability than pure data/config tasks.

**Audit result handling**:
- Any ❌ NOT DELIVERED → record in retrospective §2 Misses (🟡 painful
  at minimum; 🔴 blocking if the false `[x]` corresponds to an anchor
  in Check 9). Non-blocking for Overall Decision UNLESS paired with
  Check 9 ❌ on the same anchor.
- ≥2 false `[x]` in one sample → systemic signal. Add retrospective
  §6 promote candidate: "task self-marking unreliable in this project;
  apply-phase review must sample-audit `[x]` against spec THEN clauses."

## 9. Acceptance Anchor Coverage (external requirement traceability)

<!--
  Traces implementation back to externally-declared acceptance criteria
  (e.g. a PRD/RPD in docs/), not just to OpenSpec's own artifacts.

  AUTHORITY SOURCE: openspec/config.yaml's `acceptance_doc` section is
  the single source of truth for anchor IDs and verbatim statements.
  execution-contract.md's `## Acceptance Anchors` only lists which IDs
  this change covers (by reference, no statement copy) — the statement
  text is never duplicated, so it cannot be silently softened in a
  change artifact.

  This check breaks the self-attestation loop: tasks.md `[x]` only
  proves the checkbox moved; Check 9 proves the acceptance statement
  was actually satisfied by a real passing test, not a skipped or
  weakly-asserted one.
-->

**Step 0 — milestone-mandated anchors (declarative, NOT opt-in)**:
Read this change's proposal.md. Extract the milestone declaration
(e.g. "Milestone: R5"). If proposal declares no milestone AND config
has no acceptance_doc → this step is N/A (fall back to Step 1 opt-in
path). If config has acceptance_doc but proposal declares no milestone →
that itself is a FAIL (apply PRECHECK should have caught it; if it
slipped through, fail here).

Read openspec/config.yaml's `acceptance_doc.anchors`. Filter to those
whose `milestone` field matches the proposal's declared milestone
(support multiple milestones — union the sets). This filtered set is
the **MANDATED anchor set** for this change — every anchor in it MUST
be traced by Check 9, regardless of whether execution-contract's
Acceptance Anchors table lists it.

| Milestone (from proposal) | Mandated anchor IDs (from config filter) |
|---|---|
| <milestone> | <ID list, e.g. V5, V6, V7> |

<!--
  RATIONALE: the project declared milestone→anchor mapping at config
  level (objective fact). A change claiming that milestone inherits
  those anchors mandatorily — it cannot opt out by simply not listing
  them in its contract. This closes the opt-in evasion: under the old
  Step-2-only logic, a change could "not declare coverage" of an anchor
  and get away with a non-blocking warning; now milestone-mandated
  anchors are forced into scope.
-->

**Step 1 — read the authority source**: Read `openspec/config.yaml`.
If it has no `acceptance_doc:` key, write "N/A — no external acceptance
doc declared" and skip this check (legitimate for library projects).

**Step 2 — for each anchor in (contract-declared ∪ milestone-mandated)**:
Fetch the verbatim statement from config.yaml by ID, then verify the
implementation actually satisfies it. The traced set is the UNION of:
  (a) anchors listed in this change's execution-contract.md
      `## Acceptance Anchors` table (opt-in declarations), AND
  (b) anchors mandated by Step 0 (milestone mapping).
If an anchor is in (b) but NOT in (a) → it is ❌ NOT DECLARED (a
stronger verdict than the old "undeclared warning" — see Step 3).

| Anchor ID | Acceptance statement (verbatim from config.yaml) | Source (contract / mandated) | Planned coverage (specs/tasks) | Actual evidence (file:line + assertion verbatim) | Verdict |
|---|---|---|---|---|---|
| — | — | — | — | — | ✓ COVERED / ⚠️ PARTIAL / ❌ NOT COVERED / ❌ NOT DECLARED |

<!--
  If this change has no execution-contract.md at all (e.g. a legacy
  change archived before contracts were mandatory), the contract-declared
  set is empty and the traced set = milestone-mandated set only. Every
  mandated anchor then resolves to ❌ NOT DECLARED → FAIL. This is the
  correct outcome for legacy changes that claimed a milestone but never
  traced its acceptance anchors.
-->

**Step 3 — undeclared anchor handling (revised by mandate status)**:

| Anchor status | Verdict | Blocking? |
|---|---|---|
| milestone-mandated but absent from contract | ❌ NOT DECLARED | ✅ YES (FAIL) |
| NOT milestone-mandated and absent from contract | WARNING | no |

<!--
  CHANGE FROM v1: previously ALL undeclared anchors were non-blocking
  warnings. This let a change escape coverage by simply not listing an
  anchor in its contract (opt-in evasion). Now: if the milestone mapping
  says this change should cover it, not declaring it is a blocking FAIL.
  Non-mandated anchors remain non-blocking warnings (a change may
  legitimately not touch unrelated milestones).
-->

<!--
  RESIDUAL RISK (prompt-layer boundary): Step 0 only fires if the
  verifier actually reads proposal.md AND config.yaml. A non-cooperative
  LLM that skips reading cannot be forced by schema text alone — that
  requires harness-level command-execution receipts, out of scope here.
  Mitigation in scope: apply's proposal-milestone PRECHECK forces
  proposal to declare a milestone before apply starts, so Step 0 has
  input to read at verify.
-->

**Evidence requirements** (what counts as "actual evidence"):
- A passing automated test (e2e / integration) that exercises the FULL
  acceptance path described in the statement, with assertions on the
  statement's outcome — not just that something rendered or ran.
- OR a manual dogfood run with dated evidence (screenshot/log) attached
  to the change directory. Allowed only when automated coverage is
  genuinely impossible; record in retrospective §2 as 🟡 painful.

**Evidence citation rules** (anti-fabrication):
- The Actual evidence column MUST cite: test file path + line range
  (e.g. `tests/foo.spec.ts:33-47`) AND the exact assertion line(s)
  from that test, quoted verbatim.
- The verifier MUST open the cited file and confirm the lines exist
  and contain the cited assertions. Fabricated citations (file/line
  absent, or assertion text mismatched) = immediate FAIL + retrospective
  §2 🔴 blocking ("evidence fabrication").

**Anti-patterns that do NOT count as coverage** (mark ❌ NOT COVERED):
- tasks.md `[x]` checkbox alone (self-attestation — see Check 3)
- A test that is skipped when the system-under-test is unavailable
  (test.skip ≠ coverage — the acceptance path was not exercised)
- A test with weak assertions (element-visible / count-greater-than-zero /
  state-appeared-once) when the statement requires end-to-end correctness
- A test that exercises a SUBSET of the acceptance path (e.g. the
  statement names components A→B→C→D but the test only wires A→D)
- Recording the gap in retrospective §2 Misses (does NOT override
  Check 9 FAIL — Check 8 governs un-anchored dogfood deferrals only;
  any deferral tied to an acceptance anchor is Check 9's jurisdiction
  and FAIL cannot be downgraded by retrospective acknowledgment)

**Decision rule**:
- Any anchor ❌ NOT COVERED → Overall Decision = ❌ FAIL (blocking)
- Any anchor ❌ NOT DECLARED (milestone-mandated but absent from contract)
  → Overall Decision = ❌ FAIL (blocking)
- Anchors ⚠️ PARTIAL (subset covered) → Overall Decision =
  ⚠️ PASS WITH WARNINGS; each PARTIAL needs a retrospective §2 follow-up
